How Atlas Works

Atlas is a datacenter intelligence tool that reads cutsheet spreadsheets, loads them into Postgres, and uses an LLM to answer natural-language questions about thousands of cable connections. This course shows you how every piece fits together.

MODULE 01

What Atlas Actually Does

Before we look at any code, let's understand the product and trace what happens when you use it.

The Problem Atlas Solves

When CoreWeave builds a new data hall, thousands of cables need to connect the right devices to the right ports with the right optics. The master plan lives in a giant cutsheet spreadsheet, sometimes with 50,000+ rows.

Before Atlas, answering "how many QSFP28 optics do we still need to install?" meant manually filtering spreadsheet columns. Atlas turns that into a conversation.

📋
Upload a cutsheet
Drop your Excel file into the web UI. Atlas parses every row, normalizes the data, and loads it into a database.
💬
Ask a question
"How many SN5600 devices are in DH2?" Atlas classifies the question, pulls the right data, and hands it to an AI to answer in plain English.
🔍
Analyze a rack
Enter a room and rack number. Atlas shows every device, cable, and optic in that rack, with downloadable labels.

Trace the Journey: "How many optics do I need?"

You type that question into Atlas. Here's what happens behind the scenes in about 2 seconds:

Question → Answer Pipeline
1
Your browser sends the question
The frontend POSTs your question + auth token to /api/ask
2
Flask receives and validates
The Flask server checks your auth token, finds your site context, and decides whether to use the database path
3
Intent classification
The query router reads your question and classifies it as "optic_count" using keyword matching. No AI needed for this step.
4
SQL template executes
A pre-written SQL query runs against Postgres, counting optics by type and status
5
Results formatted for the AI
The raw database rows get compressed into a text summary that the LLM can understand, with token estimates
6
LLM writes the answer
Claude (or GPT) receives the data context and writes a natural-language answer. It can only see what the SQL returned, keeping it grounded in real data.
7
Answer arrives in your browser
The frontend displays the answer plus a badge showing whether it came from Postgres, the Rack Analyzer, or in-memory context
💡 The Big Insight: Grounded AI
Atlas never lets the AI make up SQL or guess at data. Every question gets classified into one of ~27 known types, and each type has a hand-written SQL template. The AI only sees the actual query results. This "grounded" approach means the AI can't hallucinate data that doesn't exist in your cutsheet. When you're building AI tools, this is a pattern worth stealing: let the AI write prose, but never let it write the queries.

MODULE 03

The Query Pipeline

This is the core of Atlas — how a plain-English question becomes a precise database query. Think of it like a mail sorting facility.

Step 1: Intent Classification

When your question arrives, query_intent.py runs 14 domain routers in sequence. Each one checks if the question belongs to its territory. First match wins.

query_intent.py · route_optic_intent()
def route_optic_intent(ctx: QuestionContext): optic_hits = _hits(ctx.token_set, OPTIC_WORDS) count_hits = _hits(ctx.token_set, COUNT_WORDS) if ctx.has_optic_token and count_hits: return IntentResult( "optic_count", "high", "optic token + count words", "optic", [ctx.extracted_optic] + sorted(count_hits))
PLAIN ENGLISH
Define the optic-domain router function Check if the question has optic-related words (like "optic", "transceiver") Check if it has counting words (like "how many", "count", "total") If we found a specific optic name (like QSFP28) AND count words: Return a classification result that says: Question type is "optic_count", confidence is "high" The reason: we found an optic name + counting language Domain: "optic" Signals: the optic name and the count words we matched
💡 Why Keywords Beat AI For Classification
Atlas uses keyword matching instead of an LLM for classification. Why? Speed (keyword matching takes microseconds, LLM calls take seconds), cost (no API call needed), and determinism (the same question always classifies the same way). The keyword dictionaries in query_lexicon.py are the single source of truth.

The Router Chain

The 14 routers run in a deliberate priority order. This prevents ambiguity when a question could match multiple types.

1
Upload diff/list
Highest priority. "Compare upload 5 vs upload 6" should always go to diff, even if it mentions devices.
2
Cross-site comparisons
"Optics across all sites" hits cross-site before single-site optic router.
3
Burndown / LLDP
LLDP failures and burndown queries get priority since they're operationally urgent.
4
Roles → Locations → Optics → Status → Sections
Each domain router checks its keywords. A question about "FDP spine optics" hits role first, but the role router defers to optic router when it sees optic words.
5
Trends → Data halls → Sites → Devices → IP
Lower priority catch-alls. If nothing above matched, these broader routers try.
6
General (fallback)
If no router matched, it falls back to "general" with confidence "low". The LLM gets a composite overview.

Step 2: The SQL Templates

Once the question type is known, atlas_query_router.py picks the matching SQL template. There are 27+ templates covering every question type. Here's what the optic_count one looks like:

atlas_query_router.py · optic_count template
SELECT optic_type, a_count + z_count AS cable_count, in_service, failed, pending FROM ( SELECT optic_type, SUM(CASE WHEN side='A' THEN 1 ELSE 0 END) AS a_count, SUM(CASE WHEN side='Z' THEN 1 ELSE 0 END) AS z_count, COUNT(*) FILTER (WHERE status_normalized IN ('lldp_passed','human_verified','complete')) AS in_service FROM ( -- UNION ALL of A-side and Z-side ) GROUP BY optic_type ) sub ORDER BY cable_count DESC
PLAIN ENGLISH
Give me the optic type name, and the total count (A-side + Z-side), plus how many are in-service, failed, or pending. To get this, first build a subquery that: Looks at every optic in the spreadsheet, Counts how many appear on the A-side of cables, Counts how many appear on the Z-side, Separately counts which ones have a "working" status (passed verification, human-checked, or cable-run complete) as "in_service". It checks BOTH sides independently (using UNION ALL) then groups by optic type and sorts by which optic has the most cables.
💡 A-Side and Z-Side: The Cable's Two Ends
Every cable connection has an A-side (origin) and a Z-side (destination). Each side can have a different optic. If a cable uses QSFP28 on the A-side and QSFP-DD on the Z-side, Atlas counts each optic separately. That's what the UNION ALL does: it looks at A-side optics and Z-side optics independently, then combines them. This prevents under-counting when the two ends use different optic types.

MODULE 04

The Database Layer

Atlas's database is like a warehouse with labeled shelves. Knowing the shelf layout helps you understand how questions get answered fast.

The Core Tables

Atlas stores cutsheet data across a handful of related tables. Think of it like a filing system: sites contain uploads, and uploads contain connections.

🏢
sites
One row per datacenter location. Stores the site code (ELD, QCY, DTN) and name. Everything else references back to this.
📤
cutsheet_uploads
One row per file upload. Tracks who uploaded it, when, and whether it's the "active" version. Has a soft-delete flag so old uploads stick around for trend analysis.
🔗
cutsheet_connections
The big one. One row per cable connection. Stores A-side device, port, optic, location AND Z-side equivalents, plus status and section. This is where 50,000 rows live.
🖥️
host_inventory
Device metadata from the SITE-HOSTS sheet: hostname, model, rack location, role (spine, leaf, FDP, CDU). Used for role-based queries.

The Connection Table Up Close

This is the single most important table in Atlas. Every Q&A query touches it.

atlas_schema.sql · cutsheet_connections
CREATE TABLE cutsheet_connections ( id SERIAL PRIMARY KEY, upload_id INTEGER REFERENCES cutsheet_uploads(id), site_id INTEGER REFERENCES sites(id), section TEXT, a_device TEXT, a_port TEXT, a_optic TEXT, a_model TEXT, a_loc_cab_ru TEXT, z_device TEXT, z_port TEXT, z_optic TEXT, z_model TEXT, z_loc_cab_ru TEXT, cable_id TEXT, status TEXT, status_normalized TEXT );
PLAIN ENGLISH
Create a table called cutsheet_connections with: An auto-incrementing ID for each row Which upload this row came from (links to uploads table) Which site this row belongs to (links to sites table) The topology section name (like "GRID-AGG A") A-side: device hostname, port name, optic type, device model, physical location (data hall : cabinet : rack unit) Z-side: same fields for the other end of the cable The cable's ID from the cutsheet Raw status text from the spreadsheet Cleaned-up status for fast filtering (no fuzzy matching needed)

Upload Scoping: Time-Travel for Cutsheets

Every SQL template includes this clause:

AND (%(upload_id)s::bigint IS NULL OR upload_id = %(upload_id)s::bigint)

This is clever: when upload_id is NULL, the condition is always true (you see all data for the site). When it's a specific number, you only see data from that upload. This lets Atlas answer "how did things look last week?" by querying an older upload, and "what changed?" by comparing two uploads side-by-side.

💡 Versioned Snapshots Without Complexity
This single SQL pattern gives Atlas time-travel capability without needing a separate versioning system. Each upload is a frozen snapshot of the cutsheet at that moment. The trend_status and upload_diff templates exploit this to show progress over time. When you're building your own data tools, this "scope by version ID" pattern is much simpler than trying to track individual row changes.

MODULE 05

The Clever Tricks

Atlas has some smart engineering patterns that prevent common problems. These are the patterns worth knowing so you can request them in your own projects.

Security: Three Layers Deep

Atlas handles authentication, SQL injection, and prompt injection. Each defense is independent so one failure doesn't compromise the others.

HMAC Token Auth
When you enter the PIN, Atlas creates a HMAC-signed token with a 15-minute expiry. The token is verified on every request using hmac.compare_digest(), which is timing-safe to prevent attackers from guessing the token character by character.
Parameterized SQL
All SQL uses %(param)s placeholders. User input goes through _escape_ilike() which escapes wildcards. This means if someone types '; DROP TABLE connections; -- as a device name, it's treated as literal text, not a command. The database never executes anything a user typed.
Prompt Injection Defense
Before sending data to the LLM, demo_auth_ai.py runs _sanitize_context_dict() — a recursive function that scrubs prompt injection patterns from every field. Even if someone named a device "ignore previous instructions and...", the sanitizer strips it before the LLM sees it.

Status Normalization: One Truth, Many Spellings

Cutsheets from different sites spell statuses differently. "LLDP: Passed" (two spaces), "LLDP: Pass", "Cable Is Ran Complete" all mean the same thing. Atlas normalizes them all into a small set of canonical values.

cutsheet_profiles.py · STATUS_NORMALIZATION
STATUS_NORMALIZATION = { "lldp: passed": "lldp_passed", "lldp: passed": "lldp_passed", "lldp: failed": "lldp_failed", "cable is ran complete": "complete", "cable is ran: complete": "complete", "cable not run": "not_run", "human verified": "human_verified", ... }
PLAIN ENGLISH
A dictionary that maps messy status strings: "LLDP: Passed" (with double space) → "lldp_passed" "LLDP: Passed" (single space) → same canonical value "LLDP: Failed" → "lldp_failed" "Cable Is Ran Complete" → "complete" "Cable Is Ran: Complete" (with colon) → same thing "Cable Not Run" → "not_run" "Human Verified" → "human_verified" ...and so on for every known variant. SQL queries use the normalized version for fast, exact matches.

The Three-Tier Fallback Chain

Atlas has three ways to answer a question, and it tries them in priority order. If the best source fails, it falls back to the next one. Think of it like trying to check your bank balance: first the app, then the website, then calling the phone number.

Context Priority Chain
1
Postgres (Best)
Runs a targeted SQL query. Fast, precise, handles 50K+ rows. This is the production path.
2
Normalized In-Memory (Good)
Uses the pandas-processed data from cutsheet_normalizer.py. Works without a database but hits token limits on large files.
3
Legacy In-Memory (Fallback)
Uses the old Define_Optic_Count module. Only exists for backward compatibility with small demos.

The response includes a context_source badge (POSTGRES, IN_MEMORY, RACK_ANALYZER, or EMPTY_FALLBACK) so you always know which path answered your question.

MODULE 06

The Big Picture

Let's zoom out and see how all the pieces connect, from Docker containers to Kubernetes deployment.

Two Containers, One Command

Atlas runs as two Docker containers that talk to each other. docker compose up starts both.

docker-compose.yml
services: db: image: postgres:16-alpine ports: ["9000:5432"] volumes: - ./atlas_schema.sql:/docker-entrypoint-initdb.d/ healthcheck: test: ["CMD-SHELL", "pg_isready"] web: build: . ports: ["5050:5050"] depends_on: db: { condition: service_healthy }
PLAIN ENGLISH
We're defining two services: First, a database container: Use the official Postgres 16 image (small Alpine variant) Map port 9000 on your machine to 5432 inside the container On first startup, automatically run atlas_schema.sql to create all the tables Check every 5 seconds that Postgres is healthy Second, the web app container: Build it from the Dockerfile in this folder Map port 5050 for the web UI Don't start until the database reports healthy (so the app never boots with no DB)

The Full Data Flow

Here's every major component and how data moves between them, from upload to answer.

Atlas End-to-End Architecture
📂
Excel Cutsheet (.xlsx)
The source of truth. Contains CUTSHEET tab (connections), SITE-HOSTS (device inventory), SITE-VARS (metadata), and optionally OVERHEAD (cab types).
🚪
atlas_web_app.py (Flask, port 5050)
Receives the upload. Extracts site code from filename or SITE-VARS sheet. Kicks off parallel processing.
⚙️
cutsheet_profiles.py + atlas_data_loader.py
Detects the cutsheet format (auto-detection via fingerprint scoring), normalizes column names, normalizes status strings, then bulk-inserts into Postgres (500 rows per batch).
🐘
Postgres (cutsheet_connections + views)
50,000+ rows stored with indexes on every queryable column. Materialized views (optic_inventory, device_summary) pre-compute common aggregations.
🧠
query_intent.py → atlas_query_router.py
User question classified by 14 domain routers. Matching SQL template executes against Postgres with parameterized inputs. Results formatted as compact text.
🤖
Anthropic / OpenAI API (Claude or GPT)
Receives the formatted context + conversation history. Writes a natural-language answer grounded in the actual data. Cannot see raw database or make up numbers.

What You Can Do Now

Now that you know how Atlas works, here's what you're equipped to do:

Steer AI Agents on Atlas
When asking an AI to modify Atlas, you can say "add a domain router in query_intent.py that catches questions about power distribution" instead of "make it answer PDU questions." You know the architecture.
Debug Classification Issues
When Atlas gives the wrong answer, you can trace the pipeline: was it classified wrong (check query_intent.py), did the SQL return bad data (check the template), or did the LLM misinterpret good data (check the formatted context)?
Apply These Patterns Elsewhere
Grounded AI (no LLM-generated queries), keyword classification, parameterized SQL, upload scoping, three-tier fallback chains, HMAC auth — these patterns work in any data tool you build. You now have the vocabulary to request them.
